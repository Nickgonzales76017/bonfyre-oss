# Vault Operating System

## Purpose
This vault is the operating layer for turning ideas into projects, projects into systems, and systems into offers.

## Folder Logic
- `00-Inbox`: raw capture, quick notes, loose inputs
- `01-Ideas`: opportunities worth thinking about
- `02-Projects`: active bets with a clear objective
- `03-Research`: decision-support notes
- `04-Systems`: reusable machinery and operating logic
- `05-Monetization`: offers, pricing, and sales artifacts
- `06-Logs`: execution history and feedback loops
- `Templates`: note infrastructure

## Working Rules
- an idea stays in `01-Ideas` until there is a testable execution goal
- a project belongs in `02-Projects` only when there is a specific deliverable
- a system note should describe repeatable mechanics, not aspiration
- monetization notes should sound like buyer outcomes, not internal features
- logs should capture outputs, proof, bottlenecks, and next moves

## Default Workflow
1. capture an idea
2. expand only the ideas with a plausible buyer and fast proof
3. create a project when the idea has a concrete next step
4. document the operating system behind the project
5. package successful outputs as offers
6. log what happened and fold it back into decisions

## Template Strategy
- ideas capture buyer, proof, and execution filter
- projects define success, tasks, constraints, and metrics
- systems describe inputs, outputs, flow, and bottlenecks
- research notes exist to improve decisions
- monetization notes create buyer-facing clarity
- daily logs keep the whole vault grounded in reality

## Current Priority
Use the vault to get one real service loop working, then let the richer structure support scale instead of replacing execution.
