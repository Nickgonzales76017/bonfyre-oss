Binary reference · Layer 3

Knowledge & memory.

This is the layer that turns a transcript into usable memory. Ten binaries: brief extraction, entity tagging, canonical resolution, vector embedding, SIMD similarity search, full-text and hybrid indexing, graph mapping, and a zero-copy hot-read cache. The result is an archive you can query by meaning — not by filename.

Binary reference

All 10 Layer 3 binaries.

BonfyreBrief

Structured brief extraction

Takes a clean, paragraph-structured transcript and produces a JSON brief: title, date, participants, topics covered, key decisions made, action items with owners and deadlines, unresolved questions, and a 3-sentence executive summary. The extraction uses a local summarisation model — no OpenAI API call. The brief is the canonical input for every downstream binary in Layers 4 and 5. A 45-minute transcript brief takes under 30 ms after the model is warm.

BonfyreTag

Keyword and topic tagging

Applies a two-pass tagger: first a dictionary-backed named entity pass (people, organisations, dates, products, locations, technical terms), then a topic classification pass using a lightweight classifier against a configurable taxonomy. Output is a JSON tag set attached to the brief. Tags are the primary input to BonfyreCanon for entity resolution and to BonfyreGraph for relationship mapping.

BonfyreCanon

Deduplication and canonical entity resolution

Resolves entity aliases across a corpus. If three recordings mention "the auth service," "auth-svc," and "svc-auth" — Canon resolves these to a single canonical entity. Uses a combination of string similarity, embedding proximity, and a user-editable alias dictionary. The canonical entity ID is what BonfyreGraph stores as a node — so graph queries are free of alias noise. Runs incrementally: add a new recording and only new entities need resolution.

BonfyreEmbed

Sentence embedding generation

Generates sentence-level vector embeddings from brief sections using a local embedding model (384-dimensional). Operates in batch when processing multiple transcripts — embedding 15 briefs is faster than embedding them one at a time due to batched inference. Output vectors are stored directly into BonfyreVec's LMDB index. No external embedding API. No network latency.

BonfyreVec

NEON SIMD cosine similarity and BM25 hybrid ranking

Stores embedding vectors in LMDB with a custom binary index format. Retrieval uses NEON SIMD vectorised cosine similarity — 128 dot products per instruction on ARM. On 100K stored vectors, nearest-neighbour retrieval completes in under 5 ms. Supports hybrid ranking: a BM25 keyword score is combined with the cosine similarity score to rank results that are both semantically similar and lexically relevant. Sub-linear retrieval at production scale on consumer hardware.

BonfyreIndex

Full-text and hybrid search index management

Maintains a full-text search index over brief content using a pure-C inverted index with BM25 scoring. Synced with BonfyreVec's vector index so BonfyreQuery can run hybrid searches — keyword filtering + semantic ranking in a single query. Supports incremental updates: adding a new brief updates the index in under 1 ms without a full rebuild. Index files are plain files — no running server required.

BonfyreQuery

Unified query interface across LMDB, vector, and FTS

Single API surface for all three retrieval modes: key-value lookup by artifact ID, vector similarity by embedding, and full-text search by keyword. Callers do not need to know which index backs a particular query — BonfyreQuery routes to the appropriate index and merges results. Supports compound queries: "find briefs similar to this one that also mention 'payment timeout'" (vector + keyword filter).

BonfyreKVCache

Zero-copy LMDB key-value store (512 MB map)

The hot-read path for all downstream binaries. Artifacts referenced multiple times in a pipeline run are pointer-cast directly from LMDB mmap pages — no copy into heap. Default map size is 512 MB with 4 named sub-databases (artifacts, embeddings, manifests, metadata). With bf_flatbuf, manifest fields are accessed without JSON parsing — direct struct overlay on the mmap pointer. Repeated artifact reads at near-memory speed.

BonfyreGraph

Relationship graph for entities and artifacts

Builds and queries a directed relationship graph whose nodes are canonical entities and artifacts, and whose edges are typed relationships: "mentioned-in", "related-to", "caused-by", "depends-on". Written to LMDB as an adjacency list. Graph traversal is BFS/DFS over the LMDB adjacency structure. Used by the ops recipe to automatically surface related incidents — "find all artifacts sharing 3+ entity nodes with this one."

BonfyreWeaviateIndex

Self-hosted Weaviate connector

For deployments where a full Weaviate instance is already running (common in larger team deployments), BonfyreWeaviateIndex syncs brief embeddings and structured metadata to the Weaviate schema. Supports GraphQL query passthrough. Does not replace BonfyreVec + BonfyreIndex in default deployments — it's an optional sync target for teams that already have Weaviate in their infrastructure and want Bonfyre content queryable alongside other data sources.

What comes next

Layer 3 produces the assets Layer 4 turns into publishable surfaces.

Output of Layer 3

Structured brief + entity graph + search index

A JSON brief, a tag set, canonical entity IDs, embedding vectors in LMDB, a full-text index, and a relationship graph. This is everything BonfyreRepurpose and BonfyreRender need to produce publishable output.

Input to Layer 4

BonfyreRepurpose derives multiple formats from the brief

Because the brief is a structured JSON object — not raw text — BonfyreRepurpose can derive show notes, social clips, email copy, and HTML pages in a single model pass without re-reading the original transcript.

Input to Layer 5

BonfyreOffer and BonfyreLedger consume brief context directly

The revenue layer binaries use brief fields — action items, participants, budget signals, decision outcomes — as their primary inputs for pricing, tracking, and financial reporting.
