Key capabilities

What makes this layer different from standard infrastructure.

Hot-load pipeline stages

Swap a stage without restarting the pipeline

BonfyrePipeline maintains a 32-slot plugin registry with 16 live A/B pairs. To update a processing stage, drop the new binary into the stage directory and call the hot-swap endpoint. The next artifact to arrive routes through the new code. No downtime. No rebuild. No in-flight artifacts are dropped — the swap is atomic at the function pointer level.

Fountain codes

Up to 33% packet loss is recoverable without retransmit

BonfyreDistribute delivers artifact families over QUIC with LT-code fountain coding. Any sufficient subset of received chunks reconstructs the full artifact — the receiver doesn't need every chunk, and any chunk is as useful as any other. This is why field researchers can sync archives over lossy satellite links without incomplete transfers.

OpenAI-compatible proxy

Any tool that speaks the OpenAI API uses your local model instead

BonfyreProxy exposes a /v1/chat/completions endpoint on localhost. Any existing tool configured to use OpenAI — LangChain, LlamaIndex, custom scripts — routes through BonfyreProxy to your local FPQ-compressed model instead. Zero per-token cost. No API key rotation. No rate limits. Drop-in replacement.

Binary reference

All 14 Layer 6 binaries.

BonfyrePipeline

Hot-load stage runner — 32 slots, 16 A/B pairs

The core execution environment for multi-stage pipelines. Registers stages as function pointers loaded from shared objects. Hot-swap replaces a function pointer atomically — the swap is visible to the next artifact that arrives, not the current one. Supports up to 32 stages per pipeline instance, 16 simultaneous A/B pairs for staged rollouts, and a coroutine-based execution path with per-stage channels, timeouts, and deadlines. BonfyrePipeline definitions are YAML files — stages, routing rules, failure handlers, and retry policies.

BonfyreQueue

Artifact work queue between stages

Persistent LMDB-backed FIFO queue between pipeline stages. Artifacts are enqueued with a priority, a deadline, and a payload (the BLAKE2b artifact ID). Workers dequeue and process. If a worker crashes mid-processing, the artifact is re-enqueued after a configurable visibility timeout — no work is silently lost. Supports multiple queue consumers in parallel for CPU-intensive stages. Queue state survives process restart.

BonfyreOrchestrate

Multi-pipeline coordination and state tracking

Coordinates execution across multiple BonfyrePipeline instances. Defines dependency relationships: "pipeline B starts when pipeline A produces an artifact matching this pattern." Tracks execution state across the full multi-pipeline graph: queued, in-progress, stalled, completed, failed. When a stage stalls past its deadline, BonfyreOrchestrate flags it and notifies via BonfyreTel. Used in the ops recipe to track multi-team incident resolution workflows.

BonfyreRuntime

Shared in-process coroutine execution environment

Provides the coroutine scheduler and channel primitives that BonfyrePipeline stages run on. M:N coroutine threading — many coroutines multiplexed over a small number of OS threads, with cooperative yielding at I/O boundaries. Coroutine stacks are 64 KB by default (configurable). Channels are typed, buffered or unbuffered, and select-able. This is what allows a single pipeline process to handle thousands of concurrent artifact transformations without thread-per-request overhead.

BonfyreSwarm

Peer coordination and BitTorrent v2 artifact distribution

Distributes artifact bundles peer-to-peer using BitTorrent v2 SHA-256 piece trees with 256 KB pieces. Local peer knowledge is maintained in LMDB — the swarm tracks which peers have which pieces locally without a central tracker. Gate and Meter hooks are baked in: access to swarm content can be quota-guarded. Used to distribute large model bundles and artifact archives to distributed team members without a CDN.

BonfyreDistribute

Fountain-code QUIC artifact distribution

Delivers artifact families over QUIC with LT-code fountain coding. Artifact family members multiplex over separate QUIC streams with surface-first priority — the most-needed artifact in a family gets the first available bandwidth. Reconnect logic resumes interrupted transfers using the manifest to identify already-received chunks. Up to 33% packet loss is recoverable without a full retransmit. Used for field sync over lossy connections and for distributing FPQ model shards to edge nodes.

BonfyreFPQ

4× model compression with 99.97% cosine parity

See the full FPQ internals on docs.html. Key facts: FPQ = Fractional Polynomial Quantisation — FP32 tensors quantised to INT8 with polynomial correction coefficients preserving 99.97% cosine similarity. SIMD BP128 decodes 128 E8 INT7 values per instruction block. HE-SLI adds depth-1 homomorphic encrypted linear scoring. Four quantisation levels: FPQ-4 through FPQ-1 at 4×, 6×, 8×, and 10× compression. 15 published models on HuggingFace.

BonfyreAPI

OpenAPI surface, token auth, and structured logging

An OpenAPI 3.1-compliant HTTP API surface over the full pipeline. Generates an OpenAPI spec from the BonfyreCLI command registry — every CLI command is automatically available as an API endpoint. Token auth via BLAKE2b-256 tokens. Structured JSON request/response logging with per-request trace IDs. Rate limiting hooks into BonfyreMeter and BonfyreGate. Used to expose Bonfyre pipelines as API endpoints to external tools or internal services.

BonfyreProxy

OpenAI-compatible proxy with local model routing

Runs on localhost:8080 by default. Exposes /v1/chat/completions, /v1/completions, and /v1/embeddings endpoints matching the OpenAI API schema. Incoming requests are routed to your local FPQ-compressed model via BonfyreFPQ. Response format is identical to OpenAI — existing code using the OpenAI SDK requires only a base URL change. Supports model aliases: "model": "gpt-4" routes to whatever local model you've aliased to that name.

BonfyreGate

Request gating, rate limiting, and policy enforcement

Sits in front of BonfyreAPI and BonfyreProxy. Reads quota state from BonfyreMeter and applies configurable policies: rate limits per token, per account, or per endpoint; quota enforcement (block requests when meter quota is exceeded); IP-based allow/deny lists; and time-of-day access windows. Policy rules are YAML. No per-rule compute — all rules are compiled to a Hyperscan pattern set at startup for >1 GB/s evaluation throughput.

BonfyreAuth

BLAKE2b + Ed25519 auth and token management

Issues and validates BLAKE2b-256 API tokens. Generates and stores Ed25519 keypairs for BonfyreProof bundle signing. Manages token rotation, expiry, and revocation in LMDB. No external auth service. No JWT library. Token validation is a single BLAKE2b hash comparison — under 1 µs. Ed25519 signature verification is ~50 µs. The entire auth system is a ~40 KB binary.

BonfyreMFADict

MFA dictionary and credential store

Stores and validates second-factor credentials: TOTP secrets (RFC 6238 compatible), backup codes, and WebAuthn credential IDs. TOTP validation uses constant-time comparison with a 1-step clock drift window. Backup codes are single-use and stored as BLAKE2b hashes — not plaintext. Integrates with BonfyreAuth for two-factor API access. Used in deployment configurations where API access requires MFA.

BonfyreTel

Telemetry, structured metrics, log forwarding, and communications delivery

Dual role: (1) pipeline telemetry and (2) outbound communications delivery. Telemetry: collects per-stage timing, error rates, and throughput metrics in a ring buffer; exposes a Prometheus-compatible /metrics endpoint; and forwards structured log lines to a configurable target (file, syslog, or HTTP endpoint). Communications: sends emails and webhooks from BonfyreOutreach sequences. Delivery confirmation is tracked — a sent event and a delivery confirmation event are written to BonfyreLedger separately.

BonfyreCLI

Unified command-line interface for all 53 binaries

Single entry point: bonfyre [command] [subcommand] [flags]. Every binary in all 6 layers is accessible as a CLI command. Tab-completion via bonfyre completion [bash|zsh|fish]. Pipeline runs: bonfyre pipeline run --config podcast.yaml --input episode.mp3. Direct binary calls: bonfyre transcribe --input file.wav --output transcript.txt. Built-in help for every command. BonfyreAPI generates an OpenAPI spec from the same command registry — CLI and API share the same command definitions.

The full picture

Six layers. 53 binaries. One consistent data model.

Content addressing throughout

BLAKE2b ties all 6 layers together

Every artifact produced by every layer carries a BLAKE2b hash. This is what makes the pipeline idempotent, what BonfyreProof signs, what BonfyreSync compares for incremental delivery, and what BonfyreSwarm uses as piece tree leaves. The hash is the universal artifact identifier.

No required cloud services

Layer 6 provides every infrastructure piece locally

Auth, rate limiting, metrics, API surface, model inference, artifact distribution — all in Layer 6, all running locally. The only external service in a default Bonfyre deployment is the payment processor (Stripe). Everything else runs on your hardware.

Start with one binary

You don't need all 6 layers to start

Most users start with Layer 1 + Layer 2 (ingest + transcribe) or Layer 2 + Layer 3 (transcribe + brief + search). Add layers as you need them. Each layer's binaries are independently callable — you're not required to use the full pipeline.
