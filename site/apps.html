What is a proof page?

Not a demo. Not a screenshot. A working page whose content was produced by a pipeline.

Every app listed here is a live GitHub Pages site with a real URL you can open right now. The difference between a proof page and a demo is that a demo shows you what something could look like — a proof page shows you what actually came out the other end of a specific pipeline run. The content wasn't mocked up. The page wasn't hand-written. A Bonfyre pipeline processed an input, a binary produced the structured output, BonfyreRender built the HTML, BonfyreEmit published it. The result is what you're reading.

Each card below lists the binaries that produced it. If you want to reproduce any page, those are the binaries to run. If a page shows a transcript, that transcript came from BonfyreTranscribe on real audio. If it shows a proof bundle, that bundle was signed by BonfyreProof with a real Ed25519 key. There's no layer of hand-editing between the pipeline output and what you see.

Recipe patterns

Ten pipeline patterns behind every live app.

Each recipe is a reusable binary chain. Mix and match to build your own pipeline.

Recipe 1

Audio → structured brief

Ingest → Transcribe → TranscriptClean → Paragraph → Brief — the foundation of every downstream pipeline. Produces a speaker-attributed, quality-scored, structured brief from any audio format.

Recipe 2

Brief → searchable archive

Brief → Tag → Canon → Embed → Vec → Index → Query — turns a single brief into a searchable entry in a LMDB-backed hybrid BM25+cosine archive. Scales to thousands of records with no external infrastructure.

Recipe 3

Brief → multi-format publishing

Brief → Repurpose → Render → Emit → Sync — one brief produces HTML, PDF, EPUB, RSS, and JSON simultaneously. BonfyreSync delivers to target path or remote host.

Recipe 4

Content → verified proof bundle

Hash → Proof → Pack → Ledger — BLAKE2b content-addressed hash → Ed25519 signature → packaged bundle → immutable ledger entry. Use for any deliverable that needs verifiable provenance.

Recipe 5

Work → priced offer

Brief → Offer → Pay → Meter → Ledger — brief describes the deliverable → BonfyreOffer generates a dynamic pricing surface → BonfyrePay handles checkout → BonfyreMeter tracks usage → BonfyreLedger records the transaction.

Recipe 6

Large model → laptop-runnable

FPQ compress → KVCache → Proxy — FPQ INT8 compression → LMDB KV cache for attention states → BonfyreProxy routes requests to the local model with an OpenAI-compatible API. $10K cloud GPU → $500 laptop.

Recipe 7

Recordings → family archive

Transcribe → TranscriptFamily → Embed → Vec → Render — cluster related recordings across speakers and time → embed for SIMD search → publish a browsable, searchable archive page.

Recipe 8

Audio stream → scored relay

MoQ → SpeechLoop → Tone → Segment — live audio relayed via QUIC + WebTransport → quality-scored in real time (88 eGeMAPSv02 features) → tone classified → speaker segments identified. No cloud inference latency.

Recipe 9

Outreach → sequenced follow-up

Brief → Outreach → Meter → Project — brief captures the engagement → BonfyreOutreach sequences follow-up messages → BonfyreMeter tracks interaction → BonfyreProject links to open action items.

Recipe 10

Pipeline → distributed execution

Pipeline → Queue → Swarm → Distribute — BonfyrePipeline hot-loads stages → BonfyreQueue buffers work → BonfyreSwarm routes tasks across peers → BonfyreDistribute fans out artifacts with fountain-code resilience.
