Why it exists

One security patch. All 53 binaries.

The core problem

53 binaries × N dependencies = N × 53 upgrade PRs

Every service that pulls in an npm or pip dependency owns that dependency. A BLAKE2b bug in a hashlib means 53 separate patches. A shared C runtime means one patch, one build, all 53 binaries updated simultaneously.

Size comparison

180 KB vs 50 MB

The entire 26-module runtime is 180 KB. A default Express.js application with enough capability to cover three of these modules pulls in roughly 50 MB of node_modules. The runtime costs less than a single compressed JPEG.

Zero-copy design

LMDB reads are pointer casts, not memcpy

bf_lmdb reads return a pointer directly into the mmap'd LMDB page. No copy. No allocation. bf_flatbuf reads are similarly zero-copy — the FlatBuffer schema validates the memory in-place. Hot-path artifact reads are allocation-free.

SIMD everywhere

Vectorised parsing across the stack

bf_json uses SIMD to scan tokens. bf_utf8 validates bytes 16 at a time. bf_base64 encodes/decodes 32 bytes per operation. bf_csv SIMD-scans for field delimiters. Throughput on structured data is measured in GB/s, not MB/s.

I/O and concurrency

How every binary handles work without threads.

bf_reactor

Per-core event loop (io_uring / kqueue)

Handles 4,096 concurrent connections per reactor instance without spawning a thread per connection. Uses io_uring on Linux and kqueue on macOS. BonfyreCMS, BonfyreAPI, BonfyreProxy, and BonfyreGate all embed the same reactor. Per-reactor mimalloc heaps eliminate allocation contention between cores.

bf_pool

Arena + slab allocator

Long-running pipelines need predictable memory. bf_pool eliminates heap fragmentation by managing memory in fixed-size arenas. Request-scoped allocations are freed in O(1) by resetting the arena — no per-object free() calls.

bf_pipeline

Stage execution engine with hot-load

32-slot plugin registry with atomic stage swap. Coroutine-based execution paths with channels and deadlines. A/B pairs allow 16 live experiments simultaneously. Swapping a pipeline stage mid-execution doesn't flush in-flight work.

bf_queue

Lock-free work queue

mpsc and spsc variants with back-pressure signalling. Stages that produce work faster than downstream can consume it don't spin or stall — they apply back-pressure, which propagates up the pipeline automatically.

Storage and data

Reading from disk costs nothing when the disk is a pointer.

bf_lmdb

Zero-copy LMDB bindings · 512 MB map

ACID-compliant. Supports 4 named sub-databases. Reads return a pointer into the mmap page — no copy, no allocation. The 512 MB default map handles 40 M+ short key-value records before a resize is needed.

bf_mmap

Memory-mapped file I/O

Shared access patterns across pipeline stages. Huge page support for workloads where TLB pressure matters (large model weight files, sustained artifact access). Reduces system call overhead for large sequential reads.

bf_hashmap

Open-addressing hash table

Robin Hood probing with a 75% load factor ceiling. In-memory index for keyed artifact lookups before hitting LMDB. Avoids the overhead of a tree-based map when keys are short strings or integers.

bf_csv · bf_flatbuf

Structured data I/O

bf_csv uses SIMD to scan for field delimiters, handling streaming multi-MB files in a single pass. bf_flatbuf provides zero-copy schema-verified FlatBuffer encoding — used for artifact manifests and pipeline IPC where repeated JSON parse overhead is measurable.

Encoding and serialisation

The data path between stages.

bf_json

Arena-allocated JSON parser

SIMD token scanning with an arena-allocated AST. No malloc per token. Used in BonfyreAPI, BonfyreCMS, and anywhere a binary reads or emits structured configuration. Handles multi-MB documents without heap pressure.

bf_msgpack

MessagePack for pipeline IPC

Compact binary encoding for inter-stage messages. Smaller payloads than JSON, faster parse. Used wherever pipeline stages need to hand off structured data without the overhead of schema negotiation.

bf_lz4

LZ4 frame streaming compression

Streaming compress and decompress for the artifact store. LZ4 is chosen over zstd for latency-sensitive paths — compress/decompress at >500 MB/s so the cost is I/O bandwidth, not CPU cycles.

bf_utf8 · bf_base64

Text and encoding primitives

bf_utf8 validates and iterates codepoints 16 bytes at a time (SIMD). Used in the transcript layer wherever BonfyreTranscriptClean or BonfyreParagraph processes raw text. bf_base64 encodes/decodes 32 bytes per SIMD operation — used for token encoding and credential encoding in BonfyreAuth and BonfyreGate.

Security, networking, and config

The same crypto in every binary that touches auth or signing.

bf_crypto

BLAKE2b · XSalsa20-Poly1305 · Ed25519

Shared by BonfyreAuth, BonfyreProof, BonfyreHash, and BonfyreGate. One implementation of each primitive means one audit surface. BLAKE2b content-addresses artifacts. Ed25519 signs proof bundles. ChaCha20-Poly1305 encrypts at-rest or in-transit payloads.

bf_bloom

Space-efficient deduplication

Bloom filter in the ingest path. Before writing a new artifact to LMDB, bf_bloom rejects already-seen hashes in O(1) without a full lookup. Keeps the ingest layer from accumulating duplicate data silently.

bf_picohttpparser · bf_log

HTTP parsing and structured logging

bf_picohttpparser handles HTTP/1.1 request parsing with zero allocation overhead — used across BonfyreAPI, BonfyreCMS, and BonfyreProxy. bf_log is a non-blocking structured logger writing JSON lines into a ring buffer, so logging never blocks on slow I/O.

bf_opts · bf_conf · bf_mimalloc

Config, flags, and heap

bf_opts gives every binary a consistent flag surface. bf_conf reads TOML-like config files with environment variable overlay — no binary has its own config parsing logic. bf_mimalloc is the optional per-reactor allocator for hot paths where default malloc latency is measurable.
