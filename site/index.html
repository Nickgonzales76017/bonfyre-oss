New in the stack

Recent runtime upgrades changed what Bonfyre can honestly claim.

The recent passes added more than polish. They widened the operating model underneath the site with shared-nothing serving, hot-loaded and coroutine-friendly pipelines, zero-copy cache and manifest paths, stronger transport options, and faster decode and parse paths.

API runtime

Per-core reactor path

bf_reactor now sits on a stronger request path with shared-nothing event loops, PicoHTTPParser request decoding, and per-reactor heap arenas when mimalloc is present.

Pipeline runtime

Hot-load and coroutine stages

bf_hotload makes stages swappable, while bf_pipeline_coro adds a structured-concurrency path for in-process stage flow instead of treating everything like fork-and-wait orchestration.

Transport

QUIC family streams

bonfyre-distribute now has a QUIC send path in-tree, with family-multiplexed streams, surface-first prioritization, and 0-RTT reconnect behavior for sync-heavy flows.

P2P distribution

BitTorrent v2 swarm path

bonfyre-swarm uses SHA-256 piece trees, BonfyreIndex as the local peer map, and Gate plus Meter as the credential and accounting plane for metered artifact distribution.

Zero-copy data path

LMDB cache + FlatBuffer manifests

The stack now has a read-hot LMDB cache path and FlatBuffer artifact manifests, which makes Bonfyre feel less JSON-bound in the places where repeated manifest and artifact reads were stacking up.

FPQ and privacy

Faster decode, stranger inference

On the FPQ side, SIMD BP128 speeds up E8 block decode and the HE-SLI bridge keeps the encrypted linear-scoring idea grounded enough to talk about without pretending it is already the whole production story.

The landing page should stay fast to scan. The deeper material now lives in dedicated pages so Bonfyre can show the whole story without turning the homepage into a wall of copy.
