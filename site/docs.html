bonfyre-recipe · bonfyre-run

Pipeline registry and executor.

Two new C binaries ship with the 53. bonfyre-recipe manages a SQLite-backed content-addressed registry of pipeline recipes. bonfyre-run executes any registered recipe as a level-parallel DAG. Source in cmd/BonfyreRecipe and cmd/BonfyreRun.

bonfyre-recipe internals

SQLite WAL + FTS5 + SHA-256 content addressing

Registry at ~/.local/share/bonfyre/recipes.db. Schema: recipes table (code, name, version, json_text, hash, source, timestamps) + FTS5 virtual table for full-text search over code/name/description/stage binaries. Every insert triggers FTS sync. SHA-256 is computed inline (FIPS 180-4) — no external crypto dependency. The hash is stable across machines for the same JSON.

bonfyre-run internals

Topological level assignment + fork/waitpid parallelism

Levels assigned by iterating level = max(dep_levels) + 1 until stable — correct for diamonds and fan-out/fan-in DAGs. All stages at the same level are forked concurrently (capped by --parallel N, default 8). {input} and {out}/STAGE_ID tokens are substituted before exec. After all levels, a run-manifest.json is written with per-stage exit code and wall_ms.

bonfyre-model internals

Hash-first model dependency manager

SQLite WAL-mode store with three tables: models (id, sha256, format, size), sources (model_id, url, priority), and recipe_models (recipe_code, model_id, role). SHA-256 is the canonical model identity — source URLs are pull hints resolved in priority order. HuggingFace hf://owner/repo/file URLs are converted to plain HTTPS GET requests — no Python SDK, no token required for public models. Downloaded files are SHA-256 verified and the hash stored on first pull. FTS5 full-text search over model names and descriptions.

Source priority: swarm:// → file:// → s3:// → hf:// → https://. Add any source with bonfyre-model source add <id> <url>. SHA-256 is identical regardless of source.
