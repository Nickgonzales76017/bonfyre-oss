The problem it solves

Large models shouldn't require large budgets.

Why models are heavy

32 bits per weight × billions of weights

A large AI model is a matrix of numbers. Each weight is a 32-bit float — 4 bytes. Llama-3-70B has 70 billion of them, which is 280 GB uncompressed. That doesn't fit on any consumer hardware. It barely fits on a single professional server GPU.

What FPQ does

INT8 + polynomial correction coefficients

FPQ converts each weight to an 8-bit integer (1 byte), then stores polynomial coefficients alongside the model that correct for the non-linear error INT8 truncation introduces. When inference runs, SIMD instructions expand weights back to FP32 on the fly. The model file shrinks 4×. Quality is 99.97% of the full-precision baseline.

The practical result

$10,000 cloud GPU → $500 laptop

Llama-3-70B uncompressed requires ~140 GB VRAM — a multi-card server setup. FPQ-compressed, it fits in 35 GB, which is a current-generation M-series MacBook Pro. With KV cache compression, it fits in 6 GB VRAM — a $350 consumer GPU.

How it works

Four technical components.

Quantisation

FP32 → INT8 with polynomial correction

INT8 truncation introduces non-linear rounding error. FPQ stores polynomial correction coefficients that reverse this during inference. Cosine similarity on re-expanded vectors across the 15 published models lands at 99.97% of full-precision baseline — not a rounding error, a verified measurement.

SIMD decode

AVX2 / NEON vectorised re-expansion

INT8 weights are expanded back to FP32 during inference using AVX2 (x86-64) or NEON (ARM) batched multiply-add instructions. This happens on the fly — weights are stored compressed, expanded at the moment the layer executes. Decode overhead is ≤ 2% versus loading uncompressed FP32 weights from disk. The bottleneck becomes disk I/O, not decompression CPU time.

KV cache compression

4× more context in the same VRAM

The attention KV cache accumulates during long-context inference and is where most VRAM goes during generation. FPQ quantises the KV cache independently of model weights. A model that needed 24 GB VRAM for a long context window fits in 6 GB after KV cache compression. Llama-3-70B fits on a 6 GB laptop GPU.

Fountain coding

Erasure-resilient model distribution

Model checkpoint shards are encoded with fountain codes — 256 KB pieces. Any k-of-n subset of pieces reconstructs the full checkpoint. Tolerates up to 33% packet loss before reconstruction fails. This is what lets BonfyreDistribute deliver FPQ-compressed models peer-to-peer without a central server or CDN.

Published models

15 models on HuggingFace

FPQ-quantised checkpoints are published and downloadable today. Each model ships with a companion metadata file that BonfyreFPQ uses to verify decode fidelity before first inference. If the verification fails, inference doesn't start.

HE-SLI extension

Homomorphic encrypted linear scoring

FPQ SIMD + HE-SLI adds depth-1 homomorphic encrypted linear scoring. SIMD BP128 decodes 128 E8 INT7 values per block. Inference stays in the encrypted domain for privacy-critical flows — results are produced without ever exposing plaintext input vectors to the model host.

What it replaces

vs cloud GPU for AI inference.

Cloud GPU baseline

$10,000 server GPU to run Llama-3-70B

Full-precision Llama-3-70B requires ~140 GB VRAM. That's 4–6 A100 or H100 GPUs. A single A100 is ~$10,000. Monthly cloud GPU cost for sustained inference: $2,000–$8,000/month depending on utilisation and provider.

FPQ result

Consumer hardware, same model quality

With FPQ compression, the same model runs on a $350 consumer GPU or a current-generation MacBook Pro. Recurring cost: $0/month in GPU API fees. The hardware is already under the desk.

Privacy side-effect

No data leaves the machine

Cloud GPU inference means your input data — audio, transcripts, documents — transits a third-party API. FPQ local inference means data stays on-device. HE-SLI extends this: even the model host can't see the plaintext input.
