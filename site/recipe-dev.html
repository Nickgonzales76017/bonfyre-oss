Pipeline recipe · Developer organizations

Repos and incidents → team knowledge base.

New engineers spend their first two weeks reading READMEs, watching onboarding videos, and asking the same questions in Slack. Bonfyre ingests the repository and the incident recordings and produces a navigable, searchable, always-current knowledge layer — so onboarding takes days, not weeks.

Two pipelines, one output

The repo path and the incident path converge into one knowledge base.

Developer organisations have two primary knowledge sources: the codebase itself, and the incident history. Both are typically locked up in formats that don't answer questions. Bonfyre processes each via a separate path and merges them into a unified knowledge base.

Repo path

Source code → navigable onboarding guide

BonfyreIngest ingests the repo. BonfyreTag + BonfyreCanon extract and resolve services, APIs, patterns. BonfyreBrief distils each module. BonfyreGraph maps dependencies. BonfyreRender builds an always-current architecture guide and glossary.

Incident path

Postmortem recordings → searchable atlas

BonfyreTranscribe processes each postmortem recording. BonfyreBrief extracts root cause, timeline, actions. BonfyreGraph links related incidents. BonfyreIndex builds semantic search. BonfyreProof hashes each record for audit integrity.

Worked example — repo path

A backend service repo turned into an onboarding guide.

The scenario: a 15-service backend codebase, 3 years old, 4 engineers who know it well and 1 new hire starting Monday. Here's what Bonfyre produces from the codebase.

Step 1 — BonfyreIngest (repo)

Codebase ingested and fingerprinted

BonfyreIngest processes every source file in the repo: normalises line endings, stores BLAKE2b hashes. Future runs over the same codebase skip unchanged files — only modified or added code is re-processed. The knowledge base updates incrementally on each merge to main.

Step 2 — BonfyreTag + BonfyreCanon (repo)

15 services, their APIs, and their dependencies resolved

BonfyreTag identifies service names, API endpoints, configuration keys, shared libraries, and environment variables across the full codebase. BonfyreCanon resolves aliases — if the auth service is referred to as "auth-svc", "the auth service", and "svc-auth" in different files, Canon resolves these to a single entity.

Step 3 — BonfyreBrief (per module)

Plain-English brief generated for each service

BonfyreBrief processes each service's source and config files and produces: what this service does, what it depends on, what depends on it, what configuration it requires, known failure modes in the code comments, and where to look first when something breaks. The output is the README that nobody ever writes but every new engineer needs.

Step 4 — BonfyreGraph (dependencies)

Service dependency graph built automatically

BonfyreGraph maps the service-to-service dependency graph from the entity relationships extracted by Tag and Canon. The graph shows: which services are called by which, which share database schemas, which have circular dependencies. New engineers see the architecture before they write a single line of code.

Step 5 — BonfyreRender + BonfyreEmit (repo output)

Onboarding site generated: service map, briefs, glossary

BonfyreRender generates the full onboarding site: a landing page with the service dependency graph, individual pages per service with the generated briefs, a cross-service glossary sourced from Canon's entity list, and a "where to start" guide weighted by the dependency graph. BonfyreEmit produces JSON and Markdown exports for import into other tools.

Worked example — incident path

18 months of postmortem recordings turned into a searchable atlas.

The scenario: 24 postmortem recordings from the past 18 months, each 30–60 minutes, stored in Zoom cloud. Export them once, run Bonfyre, and they become a searchable institutional memory. Here's the pipeline.

Step 6 — BonfyreIngest + BonfyreTranscribe (batch)

24 postmortem recordings → transcripts overnight

BonfyrePipeline queues all 24 recordings. BonfyreIngest normalises each. BonfyreTranscribe processes them locally at 15× real-time — 24 sessions averaging 45 minutes each take roughly 90 minutes of compute, run overnight. All 24 transcripts produced before morning without a single API call.

Step 7 — BonfyreSegment + BonfyreBrief (incidents)

Each postmortem: root cause, timeline, actions, services named

BonfyreSegment labels speakers per postmortem. BonfyreBrief produces a structured incident record: root cause, timeline of events, affected services, mitigation steps taken, open follow-up items, and contributing factors. Every postmortem becomes a structured data record — not a human's notes.

Step 8 — BonfyreGraph (incidents)

24 incidents linked by shared services and failure patterns

BonfyreGraph builds edges between incidents that share entities: same service, same failure mode, same contributing factor. A new incident involving redis timeouts automatically surfaces the 4 prior incidents that also involved redis timeouts — without any manual tagging.

Step 9 — BonfyreProof + BonfyreIndex

Each incident record hashed and indexed for semantic search

BonfyreProof signs each incident brief with Ed25519. The hash proves this brief was generated from this exact recording at this timestamp — useful for audit purposes and for future disputes about what was said in a postmortem. BonfyreIndex builds semantic search across all 24 records.

Step 10 — BonfyreRender + BonfyreSync (atlas output)

Incident atlas published internally with graph navigation

BonfyreRender generates the incident atlas: a landing page with an incident timeline and service-incident heatmap, individual pages per incident with the structured brief and related-incident graph, and a cross-incident search interface. BonfyreSync pushes to an internal server — accessible to the whole engineering team without a SaaS login.

A codebase + 24 postmortem recordings → current architecture docs, service dependency map, structured incident atlas, semantic search, graph-linked history. All on your infrastructure.

What you end up with

After both pipelines run.

Living onboarding guide

Architecture docs that update on every merge

BonfyreIngest runs on each CI merge. BonfyreBrief and BonfyreGraph re-process changed modules. The onboarding guide is always current — it reflects the actual codebase, not a stale wiki nobody updates.

Searchable incident atlas

"Did we ever see this failure before?" has an answer

BonfyreEmbed + BonfyreVec + BonfyreIndex give engineers semantic search across 18 months of postmortems. Pattern queries work: "services that failed after redis timeout" or "incidents involving the payment service on deploy day" return relevant results immediately.

Tamper-evident record

Every incident hashed and verifiable

BonfyreProof signs each incident record. If a postmortem's conclusions are ever questioned, the hash proves this brief was generated from this exact recording. The audit trail is built into the pipeline, not retrofitted after the fact.
