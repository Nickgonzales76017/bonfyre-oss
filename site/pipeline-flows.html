Pipeline patterns

Six common flows, each starting from raw input.

Audio → packaged deliverable

Ingest · Transcribe · Brief · Proof · Offer · Pack · Distribute

Audio normalized by BonfyreIngest or BonfyreMediaPrep → HCP transcription with speaker segmentation → structured brief → BLAKE2b proof bundle → dynamic pricing → packaged artifact → QUIC distribution or swarm delivery. 5–8 ms per stage.

Recording → searchable knowledge

Transcribe · TranscriptClean · Paragraph · Embed · Vec · Index · Query

Raw recording → clean transcript → paragraphs → sentence embeddings → SIMD vector index → hybrid BM25+cosine search. BonfyreTranscriptFamily clusters related recordings into a searchable family. Everything stored in zero-copy LMDB.

Run it: bonfyre-run A2 corpus.wav

Content → published pages

Brief · Repurpose · Render · Emit · Sync

Brief generated by BonfyreBrief → BonfyreRepurpose extracts multi-format content (show notes, social clips, email copy) → BonfyreRender builds HTML → BonfyreEmit produces PDF/EPUB/JSON → BonfyreSync publishes to your target.

Run it: bonfyre-run M1 content.wav

Work → revenue

Proof · Offer · Finance · Pay · Meter · Ledger · Outreach

Content-addressed proof bundle → dynamic offer pricing via BonfyreOffer → BonfyreFinance calculates margins → BonfyrePay handles checkout → BonfyreMeter tracks usage → BonfyreLedger maintains the audit trail → BonfyreOutreach sequences follow-up.

Run it: bonfyre-run P2 deliverable/

Repo → onboarding guide

Ingest · Tag · Canon · Brief · Graph · Render · Emit

Source tree ingested → BonfyreTag extracts topics → BonfyreCanon deduplicates → BonfyreBrief generates structured overview → BonfyreGraph maps relationships → BonfyreRender produces an HTML onboarding guide → BonfyreEmit publishes it.

Run it: bonfyre-run R1 ./repo

Pipeline execution

BonfyrePipeline · Queue · Orchestrate · Runtime

BonfyrePipeline runs stages with hot-load swap (32-stage registry, 16 A/B pairs). BonfyreQueue buffers work. BonfyreOrchestrate manages multi-pipeline flows. BonfyreRuntime is the shared execution environment for in-process coroutine paths.

Content-addressed artifacts

Re-runs only do work that has actually changed.

BLAKE2b hashing

Every artifact is identified by its content, not its path

BonfyreHash generates a BLAKE2b-256 hash of each artifact at the end of each stage. The hash becomes the artifact's identity. If you re-run the pipeline tomorrow with the same input audio, every stage that produces the same output is automatically skipped — no re-transcription, no re-brief, no re-render.

Proof bundles

BonfyrePack + BonfyreProof

BonfyrePack assembles all outputs into a content-addressed archive with a BLAKE2b manifest listing every included artifact and its hash. BonfyreProof signs that manifest with an Ed25519 key. Any recipient can verify: (1) the bundle is complete, (2) it hasn't been modified since signing, (3) it came from the holder of that key pair.

Incremental rebuild

Downstream stages know what changed

Because each artifact carries its own hash, downstream stages only re-execute when an upstream hash changes. Edit the brief extraction prompt and re-run — only BonfyreBrief and everything after it re-executes. The transcript and clean steps are skipped. On a large archive, this makes iterating on upstream prompts nearly instant.

Worked example

A 45-minute podcast episode, start to finish.

Here's exactly what happens when a single audio file enters a Bonfyre pipeline. Every binary listed below is a real binary in the cmd tree. Every number was measured.

Step 1 — BonfyreMediaPrep

180 MB WAV file, cleaned up and ready

The raw recording comes in at whatever sample rate and loudness your recorder saved. MediaPrep normalises it to -14 LUFS (broadcast standard), resamples to 16 kHz mono (the format Whisper performs best on), trims leading/trailing silence, and stores a BLAKE2b hash of the cleaned file. If you re-run this pipeline tomorrow with the same recording, this step is skipped because the hash matches.

Step 2 — BonfyreTranscribe

45 minutes of audio → 8,400 words in ~3 minutes

HCP-enhanced Whisper runs locally — no Deepgram API key, no per-minute charge, no data leaving your machine. On an M2 Pro it processes roughly 15× faster than real-time (so 45 min takes ~3 min). The output is a word-by-word transcript with timestamps accurate to 50 ms. You could stop here and already have done what costs $15–$30/hour on a transcription service.

Step 3 — BonfyreSegment

Two speakers, automatically labelled

Segment runs speaker diarisation on the timestamped transcript — it identifies voice boundaries and assigns [Host] and [Guest] labels to each turn. This is the difference between a raw word dump and a readable conversation. It works on the audio fingerprint, not on voice recognition, so it's language-agnostic and needs no training data for your specific voices.

Step 4 — BonfyreTranscriptClean

47 filler words gone in 23 ms

Hyperscan runs a compiled set of patterns over the transcript at >1 GB/s — removing confirmed filler words ("um", "uh"), false starts ("I — I mean —"), repeated words ("and and"), and hallucination markers (phrases Whisper commonly invents for sections of silence or music). On this 8,400-word transcript it takes 23 ms. The cleaned transcript is hashed and stored.

Step 5 — BonfyreParagraph

Raw sentences → readable sections

Raw transcripts have no paragraph breaks — everything runs together. Paragraph groups transcript turns into logical sections using topic-transition signals (speaker switches + semantic drift between consecutive sentences). The output has section headings and readable paragraphs. This is what a human editor would spend an hour doing; it takes 180 ms here.

Step 6 — BonfyreBrief

8,400 words → structured brief in 410 ms

Brief runs a local model (via BonfyreProxy → your FPQ-compressed local model) over the cleaned, structured transcript. Output: (1) a 3-sentence episode summary, (2) 8 topic headings, (3) 5 quotable moments with speaker attribution, (4) 3 follow-up questions implicit in the conversation, (5) a speaker breakdown by percentage of the episode. This is the artifact every downstream step builds on.

Step 7 — BonfyreClips + BonfyreRepurpose

5 outputs from 1 brief

BonfyreClips extracts 3 × 60-second audio clips from the best quotable moments (with timestamps from the brief). BonfyreRepurpose generates: (1) 500-word show notes, (2) a 7-post Twitter thread, (3) a 200-word email newsletter intro, (4) a 3-bullet LinkedIn update. All five are derived from the same brief — one prompt-and-template pass per format, no separate model calls.

Step 8 — BonfyreOffer + BonfyrePay + BonfyreMeter

The episode becomes a purchasable product

BonfyreOffer generates a dynamic pricing page for the episode as a premium download (PDF transcript + audio clips + show notes bundle). BonfyrePay handles Stripe-compatible checkout. BonfyreMeter records the sale and updates the usage ledger. No Shopify, no Gumroad, no per-transaction SaaS cut beyond the payment processor.

Step 9 — BonfyreEmit + BonfyrePack

Simultaneous: RSS item, PDF, EPUB, JSON

BonfyreEmit produces four outputs in a single pass: an RSS 2.0 <item> with the show notes as description, a PDF transcript, an EPUB version of the transcript, and a structured JSON artifact containing all metadata. BonfyrePack bundles all outputs into a content-addressed archive with a BLAKE2b manifest — one file you can store, distribute, or verify.

Step 10 — BonfyreDistribute

Pushed to your host over QUIC

BonfyreDistribute picks up the pack and delivers it to your target (a GitHub Pages repo, an S3 bucket, a self-hosted server) over QUIC with fountain-code erasure tolerance — meaning up to 33% packet loss is correctable before the transfer fails. For large episode archives, BonfyreSwarm adds BitTorrent v2 peer delivery so every subscriber helps distribute to the next one.

Total time for steps 1–9 (excluding transcription): under 6 seconds on M2 Pro. Transcription (the model, not Bonfyre): ~3 minutes for a 45-minute episode.
