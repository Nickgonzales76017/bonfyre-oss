Production & publishing.

Input is a structured brief. Output is something you can hand someone. Eight binaries: multi-format content derivation, HTML page generation, PDF and EPUB publishing, proof bundle signing, audio assembly, a self-contained CMS, and sync delivery. The full distance from processed data to a live page and a signed proof binder.

What this layer produces

One brief. Multiple publish surfaces. One pipeline run.

Multi-format without multiple model calls

BonfyreRepurpose runs one pass, not four API calls

Show notes, social post copy, email newsletter segment, and a LinkedIn update require the same source context but different formats and lengths. Most workflows run four separate GPT-4 API calls — one per format. BonfyreRepurpose derives all four in a single local model pass, because the brief contains all the context each format needs.

Verified provenance

Every pack is tamper-evidently signed

BonfyrePack assembles all outputs into a content-addressed archive. BonfyreProof signs the manifest with Ed25519. Any recipient can verify: (1) this bundle came from the keyholder, (2) no file inside has been modified since signing. Legal proof bundles, journalism source packs, and institutional archive submissions all use this layer.

CMS without the stack

287 KB HTTP server with dynamic schemas

BonfyreCMS is a complete content API in 287 KB: dynamic schemas, REST CRUD, full-text search, structured logging, and token auth. No Node.js. No npm. No Redis. For pipelines that need a queryable content API — not just static pages — BonfyreCMS replaces Contentful, Directus, or Strapi without the cloud dependency.

Binary reference

All 8 Layer 4 binaries.

BonfyreRepurpose

Multi-format content extraction from briefs

Accepts a brief JSON and a target format list. In a single local model pass, produces all requested output formats: show notes (800–1200 words), social post (under 280 chars), email segment (150–200 words), blog intro, newsletter summary, and LinkedIn update. Each format uses a format-specific prompt template applied to the brief. No separate API calls per format. No re-reading the original transcript. Output files are named and written to the artifact directory for downstream pickup.

BonfyreRender

HTML page generation from structured data

Takes a brief, the clean transcript, and an optional template identifier, and generates a complete HTML page: structured header, topic sections, participant list, action items, full transcript in an expandable section, and related-artifact links sourced from BonfyreGraph. Templates are plain HTML files in a configurable directory — no template engine dependency. Used to produce all live pages in the Pages app catalog. Batch mode processes a TranscriptFamily archive into a full linked site.

BonfyreEmit

PDF, EPUB, JSON, and RSS publishing

Takes the same structured brief and produces four formats simultaneously. PDF via Cairo/Pango for layout — no wkhtmltopdf, no headless Chrome. EPUB 3.0 with navigation landmarks. JSON export for programmatic consumption by other tools. RSS 2.0 feed with full content in the item body. All four formats are produced in a single Emit run — the brief is read once, not four times. Output artifacts are content-addressed and handed to BonfyrePack.

BonfyrePack

Content-addressed artifact bundle packaging

Assembles all outputs from a pipeline run — HTML, PDF, EPUB, JSON, RSS, audio files, source transcript — into a single archive with a BLAKE2b manifest. The manifest lists every file, its hash, its origin step, and its creation timestamp. The bundle ID is derived from the manifest hash — a stable, content-based identifier for this exact pipeline output. Packed bundles are the artifact unit that BonfyreProof signs, BonfyreSync delivers, and BonfyreSwarm distributes.

BonfyreProof

BLAKE2b + Ed25519 proof bundle creation and verification

Signs a BonfyrePack manifest with an Ed25519 key. The signature covers the manifest hash, not individual files — so bundle integrity verification is a single signature check, not a per-file operation. Produces a .proof file alongside the bundle. Any downstream recipient can call bonfyre proof verify --bundle [id] to check: (1) the signature is valid for the configured key, (2) the manifest hash matches the bundle contents. Used in legal workflows, journalism, institutional archive submissions, and anywhere tamper-evidence matters.

BonfyreStitch

Multi-segment audio and content assembly

Concatenates multiple audio segments or transcript sections into a single assembled output. Used for: assembling podcast episodes from guest segments + intro + outro, building highlight compilations from BonfyreClips output, and producing chapter-segmented audio files where each chapter corresponds to a BonfyreParagraph section. Output is a content-addressed audio file with a BonfyreHash manifest entry. Supports crossfade, silence padding, and level matching between segments.

BonfyreCMS

287 KB C CMS with dynamic schemas, REST API, and full-text search

A standalone HTTP server with: dynamic content schemas definable at runtime, full CRUD REST endpoints, built-in full-text search over structured fields, token-based auth (BLAKE2b-256 tokens), structured JSON logging, and a webhook system for pipeline triggers. 287 KB binary. No npm. No Node.js. No PostgreSQL required — LMDB-backed by default, optional SQLite backend. Handles 6,000+ requests/second per core on M2 Pro. Used when a pipeline needs a queryable API surface rather than static file output.

BonfyreSync

Output delivery and target synchronisation

Delivers packed artifacts to one or more targets: local filesystem path, SSH/SFTP host, S3-compatible object store, or a BonfyreCMS endpoint. Compares BLAKE2b manifests at source and target — only changed or new files are transferred. On reconnect after an interrupted sync, continues from where it left off using the manifest as the state record. Can be combined with BonfyreDistribute for QUIC delivery or BonfyreSwarm for peer-distributed delivery of larger artifact sets.

What comes next

Layer 4 also feeds Layer 5 — the revenue surface.

Output of Layer 4

Published pages, signed bundles, delivered artifacts

HTML pages at a host, PDF packs in a delivery folder, an RSS feed at a URL, a proof bundle with signature — all produced from the brief in one run. The pipeline's public face.

Feed to Layer 5

BonfyreOffer takes the brief and generates a pricing surface

The same brief that produced a show notes page can produce a checkout page via BonfyreOffer. The revenue layer and publishing layer run in parallel from the same input — you don't have to choose between publishing and selling.

No cloud provider required

Static publish to GitHub Pages or any static host

BonfyreSync's output is a folder of static files. GitHub Pages, Netlify, any rsync target, or even a USB drive. The pipeline doesn't require a cloud account to publish — it just produces files and delivers them wherever you tell it to.
