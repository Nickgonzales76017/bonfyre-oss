Three binaries, one workflow

Registry + executor + model manager. Everything you need to run.

Before these existed, running a Bonfyre pipeline meant writing shell scripts to chain binaries, manage output directories, stage model files manually, and handle partial failures. Now you type one command.

bonfyre-recipe

The recipe registry

A SQLite-backed content-addressed registry of pipeline recipes. Every recipe is a JSON document describing a DAG of stages — which binary runs, which arguments to pass, which stages depend on which others. The registry ships with 10 built-in recipes and accepts custom JSON recipes from any path. Each recipe is stored with its SHA-256 hash so you know exactly which version ran.

bonfyre-recipe list bonfyre-recipe show A3 bonfyre-recipe add my-pipeline.json bonfyre-recipe search transcribe bonfyre-recipe validate A1 bonfyre-recipe init # write starter recipe.json
bonfyre-run

The DAG executor

Loads a recipe from the registry, assigns topological levels to every stage (stages with the same level are safe to run in parallel), validates that each binary exists on PATH, forks all stages at the same level concurrently, waits for them to finish before advancing, and writes a run-manifest.json with per-stage exit codes, wall times, and overall status. One partial failure is reported exactly, not swallowed.

bonfyre-run A1 recording.wav bonfyre-run A3 episode.wav --out /tmp/ep01 --tier pro bonfyre-run M1 interview.wav --dry-run bonfyre-run P1 deliverable/ --from-stage proof bonfyre-run R1 ./myrepo --batch repos/ bonfyre-run validate A1
bonfyre-model

The model dependency manager

A SQLite-backed content-addressed registry of AI model weights. Every recipe that touches AI stages declares a list of model IDs. bonfyre-model pull --recipe A3 resolves all dependencies for that recipe, iterates each model's source priority chain (swarm → file → s3 → HuggingFace → https), downloads to a local cache, verifies SHA-256, and marks the model ready. Built-in models are seeded on first run — no manual setup required.

bonfyre-model pull --recipe A1 # ~150 MB bonfyre-model pull --recipe A3 # full suite ~8 GB bonfyre-model list bonfyre-model show whisper-large-v3 bonfyre-model verify llama-3-8b-instruct bonfyre-model status

DAG execution model

Parallel where safe, sequential where required.

bonfyre-run doesn't just chain binaries in order. It assigns every stage a topological level, then forks all stages at the same level in parallel. Stages that depend on each other run in sequence. Stages that don't run at the same time.

Level assignment

level = max(dep levels) + 1

Every stage with no dependencies runs at level 0. A stage that depends on a level-0 stage runs at level 1. A stage that depends on two stages — one at level 1, one at level 3 — waits for the deeper one and runs at level 4. The algorithm iterates until stable, which means diamond-shaped DAGs (fan-out + fan-in) are handled correctly.

# A3 level assignment (bonfyre-run show A3) L0 ingest L1 media-prep L2 transcribe L3 clean L4 paragraph # ← parallel with tone + tag L4 tone # ← all depend on clean only L4 tag L5 brief # ← depends on paragraph L6 embed # ← parallel with render + offer L6 render L6 offer # depends on brief+tone
Variable substitution

{input} and {out}/STAGE_ID

Every stage argument supports two token types: {input} expands to the file or directory you passed to bonfyre-run. {out} expands to the run's output directory. Stage output directories — where each binary should write its results — are {out}/stage-id by convention. This means stages never need to hardcode paths, and the output tree is always predictable.

# In a recipe JSON: { "id": "transcribe", "bin": "bonfyre-transcribe", "args": ["{out}/ingest", "--out", "{out}/transcribe"], "depends_on": ["ingest"] } # Expands to (for --out /tmp/run01): bonfyre-transcribe /tmp/run01/ingest \ --out /tmp/run01/transcribe
run-manifest.json

Every run produces a signed receipt

After every execution, bonfyre-run writes run-manifest.json into the output directory. It contains: the recipe code and hash, the input path, every stage with its status (ok/failed/skipped), exit code, and wall milliseconds, and the overall run status. If a stage fails, the manifest tells you exactly which one and why.

{ "recipe_code": "A1", "recipe_hash": "6fdc11a4...", "input": "recording.wav", "out": "./bonfyre-out/A1-20260417T...", "started_at": "2026-04-17T14:22:01Z", "wall_ms": 184320.0, "status": "ok", "stages": [ {"id":"ingest", "status":"ok", "exit_code":0, "wall_ms":320}, {"id":"transcribe","status":"ok", "exit_code":0, "wall_ms":182100}, {"id":"clean", "status":"ok", "exit_code":0, "wall_ms":23}, {"id":"brief", "status":"ok", "exit_code":0, "wall_ms":877} ] }
Command reference

bonfyre-run and bonfyre-recipe commands.

bonfyre-run

Running and inspecting pipelines

# Execute a pipeline bonfyre-run A1 recording.wav bonfyre-run A3 episode.wav --out /my/dir --tier pro # Plan without executing bonfyre-run dry-run A3 episode.wav # Inspect and validate bonfyre-run show A3 bonfyre-run validate A3 # Batch: run on every file in a dir bonfyre-run A1 ignored --batch ./recordings/ # Resume a partial run bonfyre-run A3 ep.wav --out /my/dir --resume # Run only a slice of the pipeline bonfyre-run A3 ep.wav --from-stage brief --to-stage pack
--out DIROutput directory (default: ./bonfyre-out/CODE-timestamp)
--dry-runPrint execution plan, no execution
--from-stage IDSkip stages before this one
--to-stage IDStop after this stage
--resumeSkip stages whose output dir already exists
--tier TIERPass --tier to every stage that accepts it
--batch DIRRun on every file in DIR, one subdir per input
--parallel NMax concurrent stages per level (default: 8)
--quietPrint only final manifest path
--verbosePrint full stage stdout
--db FILEOverride recipe registry path
--stage-opts SPer-stage flags: "id:--flag val,id2:--flag2 val"
bonfyre-recipe

Managing the registry

# List all registered recipes bonfyre-recipe list # Show a recipe's stage chain bonfyre-recipe show A3 # Add a recipe from a JSON file bonfyre-recipe add my-pipeline.json # Seed all built-in recipes bonfyre-recipe add --builtin # Full-text search across recipes bonfyre-recipe search transcribe # Check all stage binaries are on PATH bonfyre-recipe validate A1 # Print SHA-256 hash of recipe bonfyre-recipe hash A1 # Remove a recipe bonfyre-recipe rm A1 # Write a starter recipe.json to CWD bonfyre-recipe init # DB status (count, size, path) bonfyre-recipe status

Registry DB: ~/.local/share/bonfyre/recipes.db — override with $BONFYRE_RECIPE_DB

FTS5 full-text search covers code, name, description, and stage binary names.

bonfyre-model

Managing model deps

# List all registered models bonfyre-model list # Show full record for one model bonfyre-model show whisper-large-v3 # Pull all deps for a recipe code bonfyre-model pull --recipe A3 # Pull a specific model by id bonfyre-model pull llama-3-8b-instruct # Verify cached file matches SHA-256 bonfyre-model verify llama-3-8b-instruct # Print local cache path bonfyre-model path whisper-base # Add a custom model from a manifest bonfyre-model add model-manifest.json # Remove from registry (keep cache) bonfyre-model rm llama-3-70b-instruct-fpq # Remove registry record AND cached files bonfyre-model rm llama-3-70b-instruct-fpq --purge # List all downloaded files in cache bonfyre-model ls-cache # Full-text search across model records bonfyre-model search whisper # Show/add/remove sources for a model bonfyre-model sources whisper-large-v3 bonfyre-model source add whisper-large-v3 "file:///data/models/wl3.gguf" --priority 0 # DB status (count, size, path) bonfyre-model status
--db FILEOverride model registry path (default: ~/.local/share/bonfyre/models.db)
--cache DIROverride cache directory (default: ~/.cache/bonfyre/models/)
--skip-checkSkip SHA-256 verification on pull

Registry DB: ~/.local/share/bonfyre/models.db — override with $BONFYRE_MODEL_DB. Cache root: ~/.cache/bonfyre/models/ — override with $BONFYRE_MODEL_CACHE.

Model dependencies

Every recipe that uses AI declares its models.

bonfyre-model is the third leg of the tripod. Before bonfyre-run fires any AI stage, the required weights must be present in the local cache and hash-verified. The registry maps each recipe code to the exact model IDs it needs — no guessing, no manual file management.

Source priority chain

Five sources. One retrieval order.

When you pull a model, bonfyre-model iterates its registered sources in priority order (0 = highest priority) and stops at the first success. No vendor lock-in — you choose where each model lives.

0 swarm:// P2P local swarm via bonfyre-swarm. Fastest when a peer already has the model.
1 file:// Local filesystem path. Copy from a shared NAS, external drive, or build artifact.
2 s3:// AWS S3 or any S3-compatible store (MinIO, R2) via aws s3 cp.
3 hf:// HuggingFace Hub. Resolved to a plain HTTPS range-request — no Python SDK needed.
4 https:// Any HTTPS URL. Falls back to curl or wget, whichever is on PATH.
# Override built-in source with a local copy (priority 0 = first) bonfyre-model source add whisper-large-v3 "file:///mnt/models/wl3.gguf" --priority 0 # Add a private S3 mirror bonfyre-model source add llama-3-8b-instruct "s3://my-bucket/models/llama3-8b.gguf" --priority 1 # See all sources with priorities bonfyre-model sources whisper-large-v3
Custom model manifests

Register any model from any source.

If your recipe uses a private fine-tuned model or a model not in the built-in list, register it with a JSON manifest and bonfyre-model add. Then reference it by ID in your recipe JSON just like a built-in model.

// model-manifest.json { "id": "my-finetuned-whisper", "format": "gguf", "size_mb": 800, "sources": [ { "url": "file:///data/models/finetuned-ws.gguf", "priority": 0 }, { "url": "s3://my-bucket/models/finetuned-ws.gguf", "priority": 1 } ] } # Register it bonfyre-model add model-manifest.json # Now reference it in your recipe JSON: // "models": [{"id": "my-finetuned-whisper", "role": "transcribe"}] # Pull before running bonfyre-model pull my-finetuned-whisper bonfyre-run X1 recording.wav

SHA-256 is computed on first pull and stored in the registry. Subsequent pulls skip the download if the cached file verifies. Use --skip-check to bypass SHA-256 verification (fast, but not recommended for production).

Custom recipes

Any pipeline you build can become a recipe.

Recipes are JSON documents. As long as a binary exists somewhere on your PATH, it can be a stage. Dependencies between stages define the DAG. Everything else is automatic.

Recipe JSON schema

One file. One command to run it.

{ "code": "X1", "name": "My custom pipeline", "version": "1.0.0", "description": "Ingest + transcribe + custom-analyze", "models": [ { "id": "whisper-large-v3", "role": "transcribe" }, { "id": "llama-3-8b-instruct", "role": "infer" } ], "stages": [ { "id": "ingest", "bin": "bonfyre-ingest", "args": ["{input}", "--out", "{out}/ingest"], "depends_on": [] }, { "id": "transcribe", "bin": "bonfyre-transcribe", "args": ["{out}/ingest", "--out", "{out}/transcribe"], "depends_on": ["ingest"] }, { "id": "analyze", "bin": "my-custom-analyzer", "args": ["{out}/transcribe", "--out", "{out}/analyze"], "depends_on": ["transcribe"] } ] } # Register it: bonfyre-recipe add my-pipeline.json # Pull declared model deps before running: bonfyre-model pull --recipe X1 # Run it (models already cached): bonfyre-run X1 recording.wav

The "models" field is optional for recipes that use no AI stages. When present, bonfyre-model pull --recipe X1 fetches every listed model in priority-chain order and verifies SHA-256 before bonfyre-run fires any stage. Custom model IDs must be registered with bonfyre-model add model-manifest.json first.

Getting started

Start from a template, not from scratch.

Run bonfyre-recipe init to write a recipe.json template to your current directory. It comes pre-populated with the A1 stage chain. Edit the stages, change the code and name, run bonfyre-recipe add recipe.json, and you have a named, registered, hashable pipeline.

The registry is a local SQLite file — it lives at ~/.local/share/bonfyre/recipes.db and you own it. You can back it up, version-control the JSON source files, and share recipes with your team. Anyone with the same JSON file gets the same SHA-256 hash and the same reproducible run.

# First-time setup: bonfyre-recipe add --builtin # seed all 10 recipe codes bonfyre-model list # 10 models auto-registered bonfyre-recipe status # verify # Pull models for the recipe you want to run: bonfyre-model pull --recipe A1 # ~150 MB whisper-base + silero-vad bonfyre-model pull --recipe A3 # full suite (~8 GB) # Then run any pipeline (models already cached): bonfyre-run A1 recording.wav bonfyre-run dry-run A3 episode.wav # see plan first bonfyre-run A3 episode.wav --tier pro
